# 打包与交付规范

## 适用范围
适用于所有代码修改后的打包验证、产物更新和交付说明流程。

## 规则条目

### R1: 每次修改后必须执行打包验证
**断言**：每次修改项目代码后，必须在完成就近验证后自动执行 `pnpm package:win`，确保 `release\win-unpacked\素言.exe` 与 `resources\app.asar` 更新到最新版本；交付说明中必须明确报告打包是否成功以及打包产物的更新时间。若打包失败，必须说明失败原因和下一步处理方式，不得只停留在 `pnpm build`。

### R2: 文件被占用时自动关闭应用再打包
**断言**：执行打包或同步 release 目录时，若遇到 `EBUSY`、文件被占用、资源锁定等错误，必须自动检测并关闭正在运行的应用进程（`素言.exe` 或从 `release\`、`release-next\`、`release-*\` 目录启动的 Electron 进程），然后重新执行打包流程，不得因文件被占用而交付失败或停留在中间状态。

**处理流程**：
1. 检测到文件占用错误
2. 查找并确认占用进程（进程名包含"提示词"或路径匹配 `release*\win-unpacked`）
3. 强制关闭相关进程
4. 等待 2 秒确保文件句柄释放
5. 重新执行打包/同步
6. 验证产物更新时间

**违规特征**：出现 `EBUSY` 报错后直接停止、要求用户手动关闭应用、或把产物放到 `release-v2` 等临时目录而不同步回正式 `release` 目录。

### R3: 打包产物必须同步到正式 release 目录
**断言**：最终交付的打包产物必须位于 `release\win-unpacked\` 目录中。若因文件占用等原因临时输出到其他目录（如 `release-next\`、`release-v*\`），必须在文件解锁后将 `app.asar` 和 exe 等关键产物同步回 `release\win-unpacked\`，并在交付说明中确认正式目录已更新。

### R4: 正式包必须启用基础代码保护
**断言**：执行 `pnpm package:win` 时必须对打包暂存目录中的 `dist-electron` 做 JS 混淆，关闭 production source map，启用 Electron asar integrity fuses（`enableEmbeddedAsarIntegrityValidation`、`onlyLoadAppFromAsar`），正式包禁用 DevTools；打包时生成 `app-integrity.json` 供运行时完整性校验；剪切板/AI/批量等敏感模块使用更强混淆；开发用 `dist-electron` 不得被混淆脚本直接覆盖。
**示例**：
```text
pnpm package:win
# package-win.cjs 复制 dist-electron 到 stage 后调用 scripts/obfuscate-electron.cjs
# electronFuses.enableEmbeddedAsarIntegrityValidation = true
```
**违规特征**：把未混淆明文主进程 JS 打进 release、开启 sourcemap、或正式包默认开放 DevTools。

### R5: 版本隔离，禁止覆盖已发布版本
**断言**：已发布版本（首版 `v0.1.0` / 分支 `release/0.1.0` / GitHub Release 资产）为冻结交付物，后续开发必须在 `master` 以 **更高** `package.json` version（当前开发线 `0.2.0+`）进行；新产物文件名必须包含新版本号；不得 force-push 旧标签、不得替换旧 Release 附件。策略全文见 `docs/VERSIONING.md`。
**违规特征**：在未升版本号的情况下重新打包并覆盖 `v0.1.0` 安装包；把 0.2 功能直接合进 `release/0.1.0` 且不改版本；删除已发布的 `v0.1.0` Release 资产。
