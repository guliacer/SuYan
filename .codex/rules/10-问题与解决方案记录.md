## P001 — 词库视图切换卡顿（1400ms+）

### 问题描述
点击侧边栏"分类浏览"、"标签浏览"、"参数库"三个菜单项进行切换时，界面出现明显卡顿，响应时间达到 1400-4700ms。

### 发生场景
用户在分类浏览、标签浏览、参数库三个词库视图之间来回切换时触发。素材数量越多（56 条时即可复现），卡顿越严重。

### 发生时间
2026-07-08 至 2026-07-11（多个会话持续出现）

### 关键教训（务必先读）
本问题历经多轮反复，走了大量弯路，核心教训有三条：

1. **不要凭直觉叠加"优化"机制**：中途我引入的持久挂载 + 一次性预挂载三个 + `pendingMainView` 中间态 + labels cache，反而把切换从旧版的 79-687ms 拖慢到 1400-1700ms（退化 4-5 倍）。通过日志时间线对比（数据量始终 56 条未变）才锁定是代码回退而非数据增长。

2. **改性能敏感路径必须用日志实测，不能靠猜**：我连续多次基于假设改代码都没解决，直到用 `view-switch:ready` 的 `durationMs` 日志和历史时间线对比，才看清"首次挂载 ~1500ms"和"重复切换"是两个独立成本，需要分别对治。加临时性能测量代码时又因 bug 导致白屏——测量代码务必包好错误处理，用完立即移除。

3. **移除临时代码时小心误删关联逻辑**：清理性能测量代码时，我误删了填充 `mountedLexiconViews` 的追踪 effect，导致词库视图永不挂载、整片空白（白屏）。删代码后要重新通读数据流。

**最终生效方案是四个措施的组合**（见下方解决方案），核心思路：区分"首次挂载成本"和"重复切换成本"，分别用预挂载和持久挂载消除，再用启动页浮层遮住首屏过渡。

### 影响范围
- `src/features/library/components/LibraryView.tsx` — 视图切换逻辑 `openMainView`
- `PromptLexiconWorkspace` 组件的挂载/卸载周期

### 根因分析

词库切换存在**两个相互独立的成本**，必须分开对治（这是多轮弯路后才认清的关键）：

1. **首次挂载成本 ≈ 1500ms**：`PromptLexiconWorkspace` 是重组件（14 个 useState 初始化器、十余个 useMemo、多个 useEffect），且其内部 `GridPromptGallery` 会一次性渲染筛选出的**全部** 40-50 个 tile（每个含重量级 `NsfwImage`）。首次挂载时这些 DOM 节点同步创建，阻塞主线程约 1500ms。

2. **重复切换成本**：若采用"条件渲染 + `key`"（每次卸载重挂载），则每次切换都要重付 1500ms；若持久挂载（挂载后常驻、CSS 显隐），重复切换只需 88-174ms。

日志实测（`view-switch:ready` 的 `durationMs`，数据量恒为 56 条）：

| 方案 | 首次挂载 | 重复切换 |
|------|---------|---------|
| 条件渲染 + key（每次重挂载） | 1500ms | 每次 1488-1784ms 🔴 |
| 一次性预挂载三个 | 4673ms 🔴 | 88-174ms ✅ |
| **懒加载持久挂载 + 分页 + 空闲预挂载** | 预挂载在启动页背后完成 | 88-174ms ✅ |

### 解决方案（四措施组合，全部生效）

**A. 懒加载持久挂载**：视图首次被访问时才挂载（只挂被点的那一个，避免一次性挂三个的 4673ms 卡顿），挂载后用 `mountedLexiconViews` 记录并常驻，重复切换只切 CSS `contents`/`hidden`：
```tsx
const [mountedLexiconViews, setMountedLexiconViews] = useState<Set<LibraryMainView>>(() => new Set());

// 追踪 effect：视图首次访问时加入已挂载集合（务必保留，删除会导致白屏）
useEffect(() => {
  if (mainView === "parameterLexicon" || mainView === "categoryLexicon" || mainView === "tagLexicon") {
    setMountedLexiconViews((current) => (current.has(mainView) ? current : new Set([...current, mainView])));
  }
}, [mainView]);

{mountedLexiconViews.has("parameterLexicon") ? (
  <div className={mainView === "parameterLexicon" ? "contents" : "hidden"}>
    <PromptLexiconWorkspace kind="parameters" ... />
  </div>
) : null}
```

**B. GridPromptGallery 内部分页**：首屏只渲染前 24 个 tile，滚动到底部用 IntersectionObserver 哨兵增量加载，把首次挂载 DOM 数量削减到约 1/2。

**C. 空闲时逐个预挂载**：首屏加载完成后，用 `requestIdleCallback` 在浏览器空闲时**一个一个**预挂载三个词库视图（每个间隔 120ms 让出主线程），并带 2.5s 安全兜底。用户点击时组件已就绪。切忌一次性挂三个（会卡 4673ms）。

**D. 启动页浮层过渡**：把启动页从"数据加载完就 early-return 消失"改为覆盖式浮层。数据加载完后主内容在背后渲染并执行预挂载，启动页浮层（`absolute inset-0 z-50`）遮住首页骨架屏闪现；预挂载完成（或 2.5s 兜底）后浮层消失，此时首页图片和三个词库都已就绪。

**E. `openMainView` 保持极简**：直接 `startTransition(() => setMainView(view))`，不引入 `pendingMainView` 之类的同步中间空白态（那会打断 React transition 的"后台渲染新 UI、前台保持旧 UI"机制）。

### 验证方法
1. 用历史日志定位性能拐点：对比不同日期同一操作的 `view-switch:ready` 的 `durationMs`，配合数据量（`itemCount`）排除数据增长因素。
2. 首次点击词库：预挂载生效时应在启动页背后完成，用户点击即时可见。
3. 重复切换：应稳定在 88-174ms。
4. 对比 `home`、`promptSites` 切换速度作为基准（< 60ms）。

### 预防措施
- **区分首次成本与重复成本**，分别用预挂载/持久挂载对治，而非用单一机制硬套。
- **预挂载重组件必须错开逐个进行**（`requestIdleCallback` + 间隔），一次性挂多个会同步阻塞主线程。
- `startTransition` 中不要引入同步中间空白态，让 React 自行管理过渡。
- **改性能敏感路径必用日志实测**；加临时测量代码务必包错误处理并及时移除；**删临时代码后重新通读数据流**，避免误删关联 effect（曾因此白屏）。
- 遇到"以前不卡、现在卡"，优先用日志时间线 + 旧版备份 `.codex/backups/` 对照。

## P002 — labels 去重算法 O(n²) 性能瓶颈

### 问题描述
`getPromptGroupCategoryLabels` 和 `getPromptGroupTagLabels` 函数内部使用 `labels.some()` 做去重，导致 O(n²) 复杂度。在 56 条素材（约 50 个 group）下，每次切换词库视图时这些函数被调用 4-6 次，总耗时数百毫秒。

### 发生场景
切换到分类浏览或标签浏览视图时，`mergeCategoryLexiconEntriesWithPromptGroups`、`filterPromptGroupsForCategoryMenu`、`buildPromptGroupCountByCategory` 等函数内部反复调用 labels 提取函数。

### 影响范围
- `LibraryView.tsx` 第 7134-7193 行（`getPromptGroupCategoryLabels` / `addPromptGroupCategoryLabel` / `getPromptGroupTagLabels`）
- 所有依赖这些函数的 useMemo 和工具函数

### 解决方案
```tsx
// 修复前（O(n²)）：
if (!labels.some(item => normalizeLexiconItemKey(item) === normalizeLexiconItemKey(label))) {
  labels.push(label);
}

// 修复后（O(1)）：
const seenKeys = new Set<string>();
const key = normalizeLexiconItemKey(label);
if (!seenKeys.has(key)) {
  seenKeys.add(key);
  labels.push(label);
}
```

同时在 `PromptLexiconWorkspace` 中预计算 `categoryLabelsCache` 和 `tagLabelsCache`，通过 props 传递给子组件，避免重复遍历。

### 预防措施
- 去重操作使用 `Set` 而非 `Array.some()`
- 频繁调用的纯函数结果应缓存到 `Map`，通过 props 传递共享

## P003 — groupPromptImages 重复计算

### 问题描述
`groupPromptImages(promptCards, likedImageIds)` 在 `LibraryView` 和 `PromptLexiconWorkspace` 中各自独立计算一次，对全量 promptCards 做 O(n × t log t) 的分组 + 排序操作。

### 发生场景
每次词库视图挂载时，`PromptLexiconWorkspace` 内部的 `categoryPromptGroups` useMemo 对全量 `promptCards` 重新执行 `groupPromptImages`。

### 影响范围
- `LibraryView.tsx` 第 803-805 行（`imageCountByItemId` 中重复调用）
- `PromptLexiconWorkspace` 第 3143-3145 行（`categoryPromptGroups`）

### 解决方案
在 `LibraryView` 层级提前计算 `allPromptGroups`，通过 `promptGroups` prop 传递给 `PromptLexiconWorkspace`：
```tsx
// LibraryView 中一次计算，全局复用
const allPromptGroups = useMemo(
  () => groupPromptImages(promptCards, likedImageIds),
  [likedImageIds, promptCards],
);

// PromptLexiconWorkspace 直接使用，不再重算
<PromptLexiconWorkspace promptGroups={allPromptGroups} ... />
```

### 预防措施
- 昂贵计算应提升到最高公共祖先组件，通过 props 向下传递
- 不同组件对同一数据源的相同计算应合并为一个 useMemo

```tsx
type AppTitleBarProps = {
```

- 新增组件后立即运行 `pnpm typecheck` 验证

## P006 — Electron preload 类型不匹配

### 问题描述
`electron/preload/index.ts` 中 `isWindowMaximized` 的泛型参数为 `invoke<{ maximized: boolean }>`，但 `SuyanApi` 类型定义要求返回 `Promise<IpcResult<{ maximized: boolean }>>`，导致 Electron 层 TypeScript 编译失败。

### 解决方案
```tsx
// 修复前
isWindowMaximized: () => invoke<{ maximized: boolean }>(...)

// 修复后
isWindowMaximized: () => invoke<IpcResult<{ maximized: boolean }>>(...)
```
并导入 `IpcResult` 类型。

### 预防措施
- preload 中的 `invoke<T>` 泛型参数 T 必须与 `SuyanApi` 类型定义中的返回类型完全一致
- 新增 IPC 方法后同时运行 `pnpm typecheck:renderer` 和 `pnpm typecheck:electron`

## P007 — 日志系统无级别区分且路径分散

### 问题描述
旧版 `startupLog.ts` 只有一个 `logStartupEvent` 函数，无日志级别（INFO/WARN/ERROR/DEBUG）概念。日志文件位于 `library/startup.log`，命名暗示"启动日志"但实际记录了全生命周期事件。

### 影响范围
所有调用 `logStartupEvent` 的 18+ 个位置

### 解决方案
创建 `electron/main/appLogger.ts` 模块：
- 4 个日志级别：DEBUG（开发环境）、INFO（正常流程）、WARN（值得注意）、ERROR（操作失败）
- 统一存储路径：`{userData}/logs/app.log`
- 按大小轮转（1MB，保留 3 个文件）
- 敏感信息自动脱敏
- 旧版 `logStartupEvent` 保留为兼容层

### 预防措施
- 新代码使用 `logger.info/warn/error/debug(module, event, details)` 而非 `logStartupEvent`
- 参阅规则 `09-日志管理规范.md`

## P008 — IPC 错误不写入日志文件

### 问题描述
`handleResult` 中的 `catch` 将错误通过 `toErrorPayload` 返回给渲染进程，但不会写入日志文件。只有慢 IPC（≥120ms）才记录到日志。用户遇到错误时日志中找不到对应记录。

### 解决方案
在 `handleResult` 的 catch 分支中添加 `logger.error`：
```tsx
catch (error) {
  const payload = toErrorPayload(error);
  logger.error("ipc", "handler-error", {
    channel, code: payload.code, message: payload.message, durationMs,
  });
  return { ok: false, error: payload };
}
```

### 预防措施
- 所有 catch 分支应记录 ERROR 级别日志，包含错误代码、消息和上下文
- 参阅规则 09-R6：错误日志必须包含足够的上下文

### 问题描述

```tsx
// 启动时跳过同步
```

- 参阅规则 04-R6：启动路径不得执行全量派生同步
- 重任务必须由用户触发，或放到可取消的后台任务中

## P011 — CSS column-count 瀑布流布局顺序错误

### 问题描述
使用 CSS `column-count` / 多列排版做瀑布流，导致图片先填满第一列再进入下一列，少量结果全部堆在左侧。

### 解决方案
使用 JavaScript 计算的顶边对齐分列：
```tsx
const columns = distributeItemsByTopEdge(items, columnCount);
return (
  <div className="grid items-start gap-4">
    {columns.map(column => <div className="grid content-start gap-4">...</div>)}
  </div>
);
```

## P027 — 即梦（Dreamina）改版 CSR+签名墙导致网络链接导入失败

### 问题描述
导入即梦分享链接一直提示「即梦AI 链接解析失败，请检查网络后重试。」（错误码 `PROMPT_SHARE_IMPORT_FAILED`）。日志显示 `jimeng-import` 连续 6 次 `fetch:attempt` 全部 `hasRouterData:false`、`parsed:false`，且每次 `htmlLen` 稳定在 ~31KB。

- `electron/main/library/remoteMaterialDownload.ts`：**视频/封面 URL 带 `x-expires` 签名会过期**，链接模式（仅保存链接）延迟下载时旧 URL 几乎必然失效，因此对即梦作品先用隐藏窗口重新抓取新鲜签名 URL 再下载，并改用 `writeImportMediaBuffer` 支持视频落盘。

- 面对字节系（Dreamina/即梦/抖音等）带签名接口，不要尝试在主进程复现签名或存旧接口 URL 复用；用隐藏 BrowserWindow 让页面自签是唯一稳的路。

## P028 — 启动期间全量提示词识别阻塞首屏 15 秒

### 问题描述
应用启动后窗口虽已显示，但界面长时间卡住不可交互。启动日志显示 `window:show` 在 2232ms，而 `renderer:ready` 直到 15338ms 才出现，主线程被阻塞约 13 秒，违反 R6「启动路径不得执行全量派生同步」。

### 发生场景
2026-07-15 启动 Windows 打包版应用，library.json 含 207 条素材时。

### 发生时间
2026-07-15

### 影响范围
- `src/hooks/usePromptCards.ts` — `usePromptCards` 对全量 items 同步 `map(toCachedPromptCardData)`
- `src/features/library/utils/promptFilters.ts` — `resolvePromptCategory` 在卡片渲染路径调用 `suggestPromptCategories`
- `src/features/library/utils/promptAnalysis.ts` — `suggestPromptCategories` 内部调用重分析 `splitPromptToTemplate`

### 根因分析

根因是**两层叠加的同步重计算**：

1. **全量识别**：[usePromptCards.ts#L30](file:///w:/提示词/src/hooks/usePromptCards.ts) 在组件顶部被调用（[LibraryView.tsx#L763](file:///w:/提示词/src/features/library/components/LibraryView.tsx)），位于 `isInitialLibraryLoading` 早 return 之前。即使启动期间只渲染 `StartupLoadingScreen`，207 项仍被同步 `items.map(toCachedPromptCardData)` 转换，阻塞首帧。

2. **重正则分析**：其中 191 项没有显式 `category`，触发 [promptFilters.ts#L223](file:///w:/提示词/src/features/library/utils/promptFilters.ts) → `suggestPromptCategories` → `splitPromptToTemplate`。后者会对每条 prompt 跑 150+ section 的递归正则解析。

**关键陷阱**：单条 item 的转换耗时 <30ms，不触发 `measureDerivation` 的 `derivation:slow` 日志（阈值 30ms），所以 `derivation:slow` 日志为空，看上去「无慢操作」。但 207 条累加 ≈ 15s，主线程被完全锁住——这是单条阈值无法发现的累积型卡顿。

启动日志关键证据（北京时间）：
- `main:init` 86ms
- `window:show` 2232ms
- `renderer:library-load:done` 700ms（库读取本身很快）
- `renderer:ready` 15338ms（差 13 秒主线程被锁）
- `renderer:startup-screen-ready` 16527ms

### 解决方案（两措施组合，最轻量）

**A. 启动只识别首屏，剩余 idle 分批追加**（[usePromptCards.ts](file:///w:/提示词/src/hooks/usePromptCards.ts)）：

引入 `visibleCount` 状态，初始 `min(24, items.length)`——只识别首屏一屏可见的卡片。剩余在 `requestIdleCallback` 中分批（每批 64 张）追加，直到追平 `items.length`。

```ts
const FIRST_BATCH_SIZE = 24; // 首屏一屏可见的卡片数
const IDLE_BATCH_SIZE = 64; // 每次 idle 追加的卡片数

export function usePromptCards(): PromptCardData[] {
  const items = useLibraryStore((state) => state.items);
  const itemsRef = useRef(items);
  itemsRef.current = items;

  const [visibleCount, setVisibleCount] = useState(() => Math.min(FIRST_BATCH_SIZE, items.length));

  // items 长度变化（导入/删除）时校正可见数，避免越界
  useEffect(() => {
    setVisibleCount((prev) => Math.min(prev, items.length));
  }, [items.length]);

  // 首屏 24 张已转换后，剩余的在 idle 中分批追加
  useEffect(() => {
    if (visibleCount >= items.length) return;
    let cancelled = false;
    const scheduleNext = () => {
      if (cancelled) return;
      setVisibleCount((prev) => Math.min(prev + IDLE_BATCH_SIZE, itemsRef.current.length));
    };
    const handle =
      typeof window.requestIdleCallback === "function"
        ? window.requestIdleCallback(() => scheduleNext(), { timeout: 300 })
        : window.setTimeout(scheduleNext, 16);
    return () => {
      cancelled = true;
      if (typeof window.cancelIdleCallback === "function" && typeof handle === "number") {
        window.cancelIdleCallback(handle);
      } else {
        window.clearTimeout(handle as number);
      }
    };
  }, [visibleCount, items.length]);

  return useMemo(
    () => items.slice(0, visibleCount).map(toCachedPromptCardData),
    [items, visibleCount],
  );
}
```

**B. 卡片渲染路径跳过 splitPromptToTemplate 重分析**（[promptAnalysis.ts](file:///w:/提示词/src/features/library/utils/promptAnalysis.ts) + [promptFilters.ts](file:///w:/提示词/src/features/library/utils/promptFilters.ts)）：

`CategorySuggestionOptions` 新增 `skipHeavyAnalysis?: boolean`，`suggestPromptCategories` 在该标志为 true 时跳过 `splitPromptToTemplate` 调用，仅保留 8 条轻量 `categoryRules` 正则匹配（人像/插画/赛博/产品/海报/影视/艺术/图像），未命中回落到 `"未分类"`。

`resolvePromptCategory`（卡片渲染路径）传 `skipHeavyAnalysis: true`。详情页/编辑流程不传该参数，仍走完整推断，不影响用户体验。

### 验证方法
- `pnpm typecheck`：通过
- `pnpm test`：330/330 全过（含 5600 条 `toPromptCardData` 压测 <2000ms）
- `pnpm package:win`：成功，产物更新时间（北京时间）：
  - `release\win-unpacked\素言.exe` → 2026/7/15 10:31:05
  - `release\win-unpacked\resources\app.asar` → 2026/7/15 10:31:04
- 启动日志中 `renderer:ready` 应从 ~15s 降到 1-2s 量级

### 预防措施
- **启动路径绝不在组件顶部对全量数据做同步转换**——即使单条 <30ms，N 条累加会形成秒级阻塞。启动期只识别首屏可见数量，剩余 idle 分批补全。
- **`derivation:slow` 的 30ms 单条阈值无法发现累积型卡顿**，必要时应对聚合操作（如 `items.map`）整体打点而非每条打点。
- **`suggestPromptCategories` 类的重分析函数不能藏在"轻量转换"里**——`toPromptCardData` → `resolvePromptCategory` → `suggestPromptCategories` → `splitPromptToTemplate` 的链路必须可短路，启动路径传 `skipHeavyAnalysis: true`。
- **启动图库（`StartupLoadingScreen`）只依赖图片源，不依赖 `promptCards`**——所以启动期间转换 0 项也能正常显示启动页，主视图挂载后只需首屏 24 张可见卡片。
- 参阅规则 04-R6：启动路径不得执行全量派生同步（含提示词解析、分类推断）。

## P029 — 详情页按钮 tooltip 被 overflow 容器裁剪不可见

### 问题描述

2026-07-15 打开提示词详情页，鼠标悬停底部工具栏按钮（复制、导出、本地导入、识别分类等），tooltip 不显示。

### 发生时间
2026-07-15

### 根因分析
tooltip 用 `position: absolute` + `z-index: 1000` 定位，但 CSS 的 `overflow` 属性（`overflow: hidden`、`overflow-y: auto`）会裁剪所有超出该容器的子元素，**无论 z-index 多高**。详情页存在三层 overflow 裁剪：
1. `AppDialog` 的 `overflow-hidden`（弹窗外层）
2. 左栏的 `overflow-y-auto`（滚动容器）
3. 图片容器的 `overflow-hidden`

之前 P023 之前的修复（2026-07-10）只调整了工具栏 `bottom-4` → `bottom-6` 的边距，治标不治本——只要 tooltip 仍在 overflow 容器内，就会被裁剪。

### 解决方案
用 `createPortal` 将 tooltip 渲染到 `document.body`，彻底脱离任何 overflow 容器：

1. **`IconTooltipButton` 改用 Portal**：tooltip 通过 `createPortal(<span>, document.body)` 挂载到 body 末尾
2. **JS 计算位置**：`computeTooltipRect` 根据按钮的 `getBoundingClientRect()` + placement + align 计算 tooltip 的 `position: fixed` 坐标
3. **边界自适应**：tooltip 超出视口时自动翻转方向（above → below）
4. **滚动同步**：监听 scroll/resize 事件实时更新位置
5. **CSS 新增 `.icon-tooltip-button__bubble--portal`**：`position: fixed` + 进入动画，覆盖原 absolute 定位样式

### 验证方法
- `pnpm typecheck`：通过
- `pnpm test`：329/329 全过
- `pnpm package:win`：成功

### 预防措施
- **tooltip 组件必须用 Portal 渲染到 body**，不能依赖父容器不设置 overflow。任何 `position: absolute` 的浮层（tooltip、dropdown、popover）在 overflow 容器内都会被裁剪，z-index 无法解决。
- **不要用调整边距（`bottom-4` → `bottom-6`）的方式"修复" tooltip 被遮挡**——这只是把裁剪线挪了一点，根本问题是 overflow 裁剪。
- **新增 overflow 容器时，检查内部是否有 absolute 定位的浮层**——如果有，要么用 Portal，要么确保容器 `overflow: visible`。
- 参阅规则 05-UI 与组件规范：浮层组件（tooltip/popover/dropdown）统一用 Portal 渲染。

## P040 — 链接导入视频重复下载多份

### 问题描述
链接导入模式下，打开视频详情会触发下载，反复打开同一详情或详情页快速上下切换时，同一视频被重复下载多份，浪费带宽并拖慢速度。

### 影响范围
- `electron/main/library/remoteMaterialDownload.ts` — 远程素材下载
- `src/features/library/store/useLibraryStore.ts` — 渲染层下载触发

### 根因分析
去重守卫只检查 `remoteImageStatus !== "pending"`，而该状态要等整条「下载 + 写盘 + 重读库」完成才翻成 `downloaded`。视频下载慢，20+ 秒窗口内状态一直是 `pending`，任何重复触发（重复打开详情、上下切换、超时重试）都能通过守卫各自发起一份完整下载。缺的是「在途去重」而非状态检查。

### 预防措施
- 慢异步操作的去重不能只依赖「最终状态字段」，必须加在途（in-flight）去重，覆盖操作进行中的时间窗口。

## P041 — 默认关闭 GPU 导致启动/图片/视频不流畅

### 问题描述
启动、图片滚动、动画、视频播放等本可由显卡合成加速的环节，因硬件加速默认走 `stable` 模式（关闭 GPU）而全部走软件渲染，牺牲了流畅度。直接默认开启 GPU 又可能触发 Windows GPU 缓存异常白屏（当初设默认稳定模式的原因）。

### 发生场景
2026-07-16 排查「能用显卡的地方都用显卡」的整体性能优化时。

### 影响范围
- `src/features/library/types/appAcceleration.ts` — 加速设置类型与默认值
- `electron/main/app/gpuAccelerationSettings.ts` — 启动加速决策与降级持久化
- `electron/main/app/gpuCrashGuard.ts`（新增）— GPU/渲染进程崩溃恢复
- `electron/main/index.ts` — 接入崩溃守卫
- `electron/main/runtime/ffmpegEncoders.ts`（新增）— 视频硬件编码探测与参数规划
- `electron/main/batch/videoCompressor.ts`、`electron/main/library/videoImportNormalizer.ts` — 硬件编码 + 软件回退
- `src/features/library/components/PerformanceSettingsDialog.tsx` — 文案与「重新启用 GPU」入口

### 根因分析
硬件加速是全局启动决策，一旦开启无法逐环节回退；Windows 上 GPU shader 磁盘缓存损坏会导致空白画面且不可复现，因此此前保守地整体关闭 GPU。真正能用显卡的只有 Chromium 合成层（启动/图片/动画/视频播放）；AI 分析、图像反推走远程 API，sharp 图像处理是纯 CPU 库，均无本机 GPU 空间。

### 解决方案
1. 默认开启 GPU（`gpu-experimental`），启动时追加 `disable-gpu-shader-disk-cache` 精准规避白屏主因。
2. 新增崩溃守卫：90 秒窗口内 GPU/渲染进程崩溃累计达 2 次，写入 `gpuAutoDisabledByCrash` 标志并重启进入软件模式；未达阈值的单次渲染崩溃直接 reload 自愈；正常运行 5 分钟后清除历史降级标志。
3. 用户在设置里可手动「重新启用 GPU」。
4. 视频转码/压缩优先用硬件编码器（NVENC/QSV/AMF，经真实 smoke test 验证），失败自动回退软件 libx264/libx265，转码后仍走校验，画质与兼容不退化。

### 验证方法
`pnpm typecheck`、`pnpm test`、`pnpm package:win`；启动加速设置默认显示 GPU 模式；日志 `gpu:status` 反映有效加速；制造 GPU 崩溃可见 `gpu:auto-downgrade` 并重启进入软件模式。

### 预防措施
- 能用显卡的只有渲染合成层，远程 AI 与 sharp/CPU 库不要臆想 GPU 加速。
- 默认开启风险性硬件能力时，必须配套自动降级 + 手动恢复，不把不可复现故障抛给用户。
- 硬件编码器可用性依赖具体机器，必须运行时 smoke test 探测并保留软件兜底。

## P042 — 正式包防拆包基础保护（混淆 / asar fuse / 关 DevTools）

### 问题
Electron 默认 `app.asar` 可被直接解包，主进程与 preload 为可读 JS，正式包也易打开 DevTools，导致软件被低成本抄袭换皮。

4. 正式包 `webPreferences.devTools = false`，并拦截 F12 / Ctrl+Shift+I 等快捷键。
5. 安装包补充 copyright、legalTrademarks 与固定安装包命名。

### 注意
开发目录 `dist-electron` 保持明文，便于本地调试；保护只作用于打包产物。

## P044 — 词库浏览卡片底部被固定高度限制

### 问题描述
分类浏览、标签浏览和参数词库的主卡片被限制在当前视口内，底部内容只能在卡片内部滚动，视觉上像卡片底部被截断，无法随页面继续向下查看。

### 发生场景
2026-07-17 打开分类浏览、标签浏览或参数词库，右侧提示词组或参数集合数量较多时。

### 影响范围
- `src/features/library/components/LibraryView.tsx` — `LexiconSection`
- `src/features/library/components/LibraryView.tsx` — `ParameterLexiconExplorer`
- `src/features/library/components/LibraryView.tsx` — `CategoryPromptGroupExplorer`
- `src/features/library/components/LibraryView.tsx` — `TagPromptGroupExplorer`

### 根因分析
词库主卡片外层使用 `max-h-[calc(100vh-14rem)]` 和 `overflow-hidden`，内部 Explorer 又依赖 `h-full`、`flex-1` 与独立 `overflow-y-auto`。主页面场景下这些约束叠加后，内容被固定在一块内部滚动区域里，无法使用页面主滚动自然延展。

### 解决方案
为词库卡片和 Explorer 增加 `page` / `bounded` 两种布局模式：主页面使用 `page` 布局取消外层高度上限和内部内容滚动限制，让页面主滚动承接长内容；弹窗继续使用默认 `bounded` 布局，保持原有受限滚动。左侧菜单在主页面保留最大高度和独立滚动，避免超长菜单挤压右侧卡片。

### 验证方法
运行 `pnpm typecheck`、`pnpm test` 和 `pnpm package:win`；手动打开分类浏览、标签浏览和参数词库，确认主卡片底部不再被视口高度截断，页面可以继续向下滚动查看更多内容。

### 预防措施
- 页面级卡片不要同时使用视口 `max-height` 与内部 `overflow-hidden` 包裹主要内容。
- 弹窗和页面复用同一组件时，应显式区分受限滚动布局与自然页面布局。

## P048 — 启动页轮播后层图像透过前层图像

### 问题描述
启动页轮播的侧边图像叠放时，后面一张图像会从前面一张图像的重叠区域透出来，看起来像后层图压在了前层图上。

### 发生场景
2026-07-17 启动应用时，启动图库横向轮播中左右侧图片发生重叠。

### 影响范围
- `src/features/library/components/LibraryView.tsx` — `StartupLoadingScreen` 启动页轮播层级计算
- `src/styles/tokens.css` — `.startup-slide` 启动页卡片透明与暗化样式

### 根因分析
轮播卡片将透明度直接设置在整个 `figure` 父级上，父级透明会让底下层级更低的图像透过当前卡片显示。即使 z-index 按距离中心排序正确，视觉上仍会出现后层图像穿透前层图像的问题。

### 解决方案

### 验证方法
运行 `pnpm typecheck` 和 `pnpm package:win`；启动应用，确认左右两侧重叠区域中后层图片被前层图片遮挡，只保留前层图片的暗化效果。

### 预防措施
- 需要真实遮挡关系的叠层图片，不要用父级 `opacity` 做远近变暗。
- 叠层轮播应把层级排序和明暗表现拆开：z-index 控制遮挡，内部蒙层或滤镜控制视觉弱化。

### 根因分析
素材浏览在完整排序前先由 `usePromptCards` 对原始 items 做分批切片，再由页面做第二层可见窗口分页。缺失项如果位于初始切片或页面窗口之外，就会暂时或持续不可见，而批量管理的窗口更大，因此两边表现不同。同时，渲染状态的对象复用比较遗漏了 `mediaStorage`，仅状态、大小或修改时间变化时可能错误复用旧对象，丢掉最新的 `missing` 健康状态。

### 解决方案
分批转换、素材浏览排序和批量管理分组均稳定地把 `missing` 条目放在可用条目前面；同一图片提示词组在没有精选或真实视频优先级时使用缺失项作为主项，确保占位状态直接可见。将素材条目身份比较提取为纯函数并完整比较 `mediaStorage` 的 root、相对路径、大小、修改时间和状态，健康变化不再复用旧对象。

### 验证方法
运行 `pnpm typecheck`、相关 Vitest 单测、完整 `pnpm test` 和 `pnpm package:win`；删除外链源文件后重新扫描并重启，素材浏览和批量管理都应在前部显示同一“源文件缺失”条目，手动删除索引不影响其他文件。

### 预防措施
- 性能分批必须先处理需要用户关注的异常状态，不能让异常项长期落在不可见窗口外。
- 用对象身份优化大列表时，比较字段必须覆盖所有会改变渲染结果的数据，尤其是外链路径和健康状态。
- 聚合卡片需要为缺失媒体提供明确主项或可见状态，不能只依赖用户进入详情后发现。
