# 日志管理规范

## 适用范围
适用于 `W:\提示词` 项目内所有主进程日志、渲染进程日志桥接、日志文件存储、轮转、导出和安全过滤。

## 规则条目

### R1: 正式包日志必须跟随软件目录
**断言**：Windows 安装版日志必须位于安装目录下的 `logs\`，免安装目录版和单文件便携版日志必须位于便携版所在目录下的 `logs\`；开发模式日志仍位于 `%APPDATA%\SuYan\logs\`，避免污染源码目录。旧 `%APPDATA%\SuYan\logs\` 和 `library/startup.log` 由迁移函数并入新版正式包日志目录，并删除已成功迁移的旧文件。新代码不得在其他位置创建日志文件。
**示例**：
```text
D:\Apps\SuYan\logs\app.log          ← 安装到 D 盘时的当前日志
E:\Portable\SuYan\logs\app.log      ← 便携版当前日志
<软件目录>\logs\app.log.1             ← 上一轮日志
<软件目录>\logs\app.log.2             ← 最早的日志
```
**违规特征**：正式包继续写入 `%APPDATA%`、便携版写入系统临时解压目录，或在 `library/`、项目根目录及其他路径下创建日志文件。

### R2: 日志条目必须使用统一 JSON 格式
**断言**：每条日志为一行 JSON（NDJSON），必须包含以下字段：`at`（ISO 时间戳）、`elapsedMs`（距进程启动毫秒数）、`level`（日志级别）、`module`（模块标识）、`event`（事件名称）。错误日志还应包含 `code`（错误代码）。
**示例**：
```json
{"at":"2026-07-11T18:30:00.123Z","elapsedMs":1234,"level":"INFO","module":"main","event":"app:ready"}
{"at":"2026-07-11T18:30:05.456Z","elapsedMs":6567,"level":"ERROR","module":"ipc","event":"handler-error","code":"AI_REMOTE_TIMEOUT","channel":"ai:analyze-prompt","message":"AI 请求超时","durationMs":30006}
{"at":"2026-07-11T18:30:10.789Z","elapsedMs":11900,"level":"WARN","module":"ipc","event":"ipc:slow","channel":"image:import-files","durationMs":5200,"ok":true}
```
**违规特征**：使用非结构化纯文本日志、缺少 `level` 或 `module` 字段、使用 `console.log` 替代结构化日志。

### R3: 日志级别必须正确使用
**断言**：日志级别含义如下，调用方必须按语义选择：

| 级别 | 含义 | 使用场景 |
|------|------|---------|
| `DEBUG` | 调试信息 | 开发环境的详细跟踪，生产环境默认不记录 |
| `INFO` | 正常流程 | 启动事件、视图切换、操作完成等 |
| `WARN` | 值得注意 | 慢 IPC（≥120ms）、非致命异常、降级处理 |
| `ERROR` | 操作失败 | IPC handler 异常、文件读写失败、AI 请求失败 |

**示例**：
```ts
logger.info("main", "app:ready");
logger.warn("ipc", "ipc:slow", { channel, durationMs, ok });
logger.error("ipc", "handler-error", { channel, code: "AI_REMOTE_TIMEOUT", message });
logger.debug("renderer", "view-switch:click", { from: "home", to: "tagLexicon" });
```
**违规特征**：正常流程使用 `ERROR` 级别、异常场景使用 `INFO` 级别、生产环境大量输出 `DEBUG` 日志。

### R4: 模块标识必须规范化
**断言**：`module` 字段使用以下固定标识之一：

| 模块标识 | 含义 |
|----------|------|
| `main` | Electron 主进程生命周期 |
| `ipc` | IPC 通信层 |
| `renderer` | 渲染进程桥接的事件 |
| `ai` | AI 分析、优化、翻译、反推 |
| `library` | 素材库读写 |
| `network` | 代理、外部请求 |
| `window` | 窗口状态 |

**违规特征**：使用自由命名的模块标识（如 `"myFeature"`、`"temp"`），或省略模块标识。

### R5: 日志文件轮转按大小分割，最多保留 3 个文件
**断言**：单个日志文件上限 1 MB；超过时按序号轮转：`app.log` → `app.log.1` → `app.log.2`（最老）→ 删除。总日志保留量不超过 3 MB。轮转逻辑由 `appLogger.ts` 的 `rotateIfNeeded` 函数自动执行，调用方无需手动管理。
**违规特征**：修改轮转上限超过 2 MB、保留超过 5 个旧文件、手动删除或跳过轮转。

### R6: 错误日志必须包含足够的上下文
**断言**：`ERROR` 级别日志必须包含以下信息以便定位问题：
- `code`：结构化错误代码（如 `AI_REMOTE_TIMEOUT`、`LIBRARY_ITEM_NOT_FOUND`）
- `message`：人类可读的错误描述
- `channel`（IPC 错误时）：触发错误的 IPC 通道
- `durationMs`（超时错误时）：操作耗时

**示例**：
```json
{"level":"ERROR","module":"ipc","event":"handler-error","code":"AI_REMOTE_TIMEOUT","channel":"ai:analyze-prompt","message":"AI 请求超时","durationMs":30006}
```
**违规特征**：错误日志只有 `"发生错误"` 文本，没有错误代码、通道或耗时信息。

### R7: 日志必须支持用户导出
**断言**：应用必须通过 `window.suyanApi.exportLogs()` 提供日志导出功能。导出操作将所有日志文件合并为一个 `.log` 文本文件，通过系统保存对话框让用户选择存储位置。用户反馈问题时应引导其导出并附带日志文件。
**示例**：
```ts
// 渲染层调用
const result = await window.suyanApi.exportLogs();
if (result.ok && result.data.exported) {
  showStatus("日志已导出。");
}
```
**违规特征**：要求用户手动到文件系统中复制日志文件、日志导出时遗漏旧轮转文件。

### R8: 日志记录不得阻塞主流程
**断言**：`log()` 函数必须同步返回（fire-and-forget），文件写入通过异步队列串行执行，写入失败静默忽略。日志操作不得抛出异常、不得阻塞渲染进程、不得导致 IPC 延迟。
**示例**：
```ts
// 正确：同步返回，不 await
logger.info("main", "app:ready");

// 正确：内部写入队列自动串行化
log("INFO", "main", "event-a");
log("INFO", "main", "event-b"); // 保证 b 在 a 之后写入
```
**违规特征**：`await log(...)`、在 `log` 返回前执行文件 I/O 同步操作、日志写入失败导致功能异常。

### R9: 日志中不得包含敏感信息
**断言**：日志写入前必须经过 `sanitizeDetails` 脱敏处理。以下内容不得出现在日志中：
- API Key、密码、Token、授权凭证（匹配关键词自动替换为 `***`）
- 用户提示词全文（超过 500 字符自动截断）
- 图片文件的二进制内容或 base64 编码
- 用户个人数据（邮箱、手机号等）

**自动脱敏关键词**：`apikey`、`api_key`、`api-key`、`secret`、`password`、`token`、`authorization`、`credential`（不区分大小写）。
**示例**：
```ts
// 输入
logger.info("ai", "settings-save", { apiKey: "sk-abc123", model: "gpt-4" });
// 写入文件
{"level":"INFO","module":"ai","event":"settings-save","apiKey":"***","model":"gpt-4"}
```
**违规特征**：日志中出现 `sk-`、`Bearer `、密码明文、完整提示词超过 500 字符。

### R10: 新代码必须使用 `logger` 而非 `logStartupEvent`
**断言**：新增的日志调用必须使用 `import { logger } from "./appLogger"` 并调用 `logger.info(module, event, details)` 等方法。旧的 `logStartupEvent` 保留为兼容层，现有调用无需立即迁移，但新代码不得使用。
**示例**：
```ts
// 新代码（推荐）
import { logger } from "./appLogger";
logger.info("library", "item-saved", { itemId, durationMs });

// 旧代码（兼容，无需修改）
import { logStartupEvent } from "./startupLog";
logStartupEvent("app:ready");
```
**违规特征**：新增代码中使用 `logStartupEvent`，或直接使用 `console.log` / `console.error`。

## 日志查看方法

### 方法一：应用内导出（推荐）
在应用中调用日志导出功能，系统会弹出保存对话框，将所有日志文件合并为一个 `.log` 文件。

### 方法二：直接查看文件
正式包日志位于软件目录的 `logs\`；开发模式日志位于 `%APPDATA%\SuYan\logs\`：
```powershell
# 在软件目录中查看正式包当前日志（PowerShell）
Get-Content -LiteralPath '.\logs\app.log' -Encoding UTF8 -Tail 50

# 查看正式包全部日志文件
Get-ChildItem -LiteralPath '.\logs\'
```

### 方法三：JSON 结构化查询
由于每行是 JSON，可以使用 PowerShell 或 jq 按条件筛选：
```powershell
# 只看 ERROR 级别
Get-Content -LiteralPath '.\logs\app.log' -Encoding UTF8 |
  ForEach-Object { $_ | ConvertFrom-Json } |
  Where-Object { $_.level -eq "ERROR" }

# 只看特定模块
Get-Content -LiteralPath '.\logs\app.log' -Encoding UTF8 |
  ForEach-Object { $_ | ConvertFrom-Json } |
  Where-Object { $_.module -eq "ipc" }
```
